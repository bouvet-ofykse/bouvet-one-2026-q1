Hva er NAIS

NAIS er en opinionated PaaS bygget oppå Kubernetes.

Bygget av NAV

Den gir deg en standardisert måte å bygge, deploye og kjøre applikasjoner på – uten at hvert team må løse alle plattformproblemene selv.

  • NAV ønsket å standardisere sine applikasjoner kjørende i sky
  • Fant ut at det de hadde laget var kult og ønsket å dele med andre offentlige instutisjoner i Norge

Hva er en opinionated PaaS?

En plattform som har tatt valg for deg

Du skal ikke måtte kunne Kubernetes

Sikkerhet skal være default riktig

Standardisering > fleksibilitet

  • Samme deploy-mønster
  • Samme logging
  • Samme metrics
  • Samme secretshåndtering
  • Plattformen optimaliserer for organisasjonen, ikke for hvert enkelt team.

Hvorfor Nais?

Fordeler

  • 🚀 Developer experience
  • 🔐 Innebygd sikkerhet
  • 📦 Standardisering
  • 📊 Observability out-of-the-box
  • ☁️ Skalerbarhet
  • 🤝 Plattform-team og community
  • 💵 Kostnad (?)
  • 👌 Nais Console
🚀 Developer experience:
  • Veldig lav terskel for deploy
  • nais.yaml + push → app i prod
  • Lite behov for å forstå underliggende Kubernetes i detalj
  • Gode defaults → “golden path”
  • En god del self service, mye man kan fikse ekelt på egen hånd
  • Utviklere kan fokusere på kode, ikke infra og DevOps-ting
🔐 Innebygd sikkerhet
  • Automatisk håndtering av: TLS, autentisering (ofte via OAuth 2.0 / OIDC), secrets (Vault-integrasjon)
  • Strenge standarder → vanskelig å gjøre “dumme ting”
  • Mindre sikkerhetsarbeid per team
📦 Standardisering
  • Felles måte å: deploye, konfigurere, monitorere, logge
  • Integrasjoner “bare funker” (f.eks. Kafka, databaser, Unleash)
  • Mindre snowflake-setup
📊 Observability out-of-the-box
  • Logging, metrics og tracing er ferdig satt opp, det eneste du trenger å gjøre er å skru det på
  • Gjør det enkelt å debugge i både test og prod
  • Vi har tatt i bruk Loki, Prometheus og Tempo
  • Kan settes opp egne boards i Grafana
  • Det aller meste er satt opp, bare skru det på
☁️ Skalerbarhet (og reparerbarhet)
  • Bygger på Kubernetes → autoscaling og robust drift
  • Podder reparerer seg selv, utrullingsstrategier, load balancing osv..
  • Håndterer mye trafikk uten at team må gjøre mye selv
  • Nyter godt av alle fordeler Kubernetes tilbyr
🤝 Plattform-team og community
  • NAIS-teamet håndterer: cluster drift, oppgraderinger, sikkerhetspatching, håndtering av namespaces og teams
  • Resultat: Mindre DevOps-ansvar i hvert team
💵 Pris(?)
  • NAV tar en fast pris per år (symbolsk)
  • NAV får en regning fra Google (GCP)
  • Denne distribueres videre til alle de ulike "kundene"
  • Resultat: Lavere kostnad pga "kvantumsrabatt" hos GCP
👌 Nais console
  • Oversikt over alt det viktigste som angår applikasjonene dine
  • Kommer en demo av denne

Ulemper

  • 🧩 Mindre fleksibilitet
  • 🪄️ Abstraksjon skjuler kompleksitet
  • 🔒 Vendor / platform lock-in
  • 🐢 Avhengig av plattform-teamet
  • 📚 Læringskurve
  • 🧪 Lokal utvikling ≠ produksjon
🧩 Mindre fleksibilitet
  • Du må følge “NAIS way”
  • Mer komplisert å gjøre ting som går utenfor standard/den opptråkkede sti
  • For eksempel mer komplekse nettverksoppsett
  • Resultat: Kan føles begrensende for avanserte behov
🪄 Abstraksjon skjuler kompleksitet
  • Du slipper å deale med Kubernetes… helt til noe går galt
  • Debugging kan bli tricky hvis du ikke forstår hva som skjer under panseret
  • Bør ha noe fundamental kjennskap til hvordan et cluster fungerer
🔒 Vendor / platform lock-in
  • Tett koblet til NAIS-økosystemet
  • Flytte ut → krever jobb
  • MEN: Du har hvertfall containerisert appen :)
  • Også locked til GCP (hvertfall foreløpig)
  • Tror Nais jobber med eller tenker på exit-strategi (som alle andre)
🐢 Avhengig av plattform-teamet
  • Hvis NAIS ikke støtter noe må du vente eller finne andre/egne løsninger
  • Noe funksjonalitet eller integrasjoner kan bli lagt til om ønsket er stort nok
📚 Læringskurve
  • Lett å komme igang
  • MEN: du må forsatt ha innsikt i:
  • cloud native arkitektur
  • (litt) yaml og infrastructure as code
  • Conteinerisering
🧪 Lokal utvikling ≠ produksjon
  • Vanskelig å fullt ut emulere NAIS lokalt
  • Krever ofte mocks eller deploy for testing
  • Eksempel: databaser, buckets, autentisering osv

En app i NAIS

NAIS


apiVersion: nais.io/v1alpha1
kind: Application
spec:
image: my-api:1.0.0
port: 8080

replicas:
min: 2
max: 5

accessPolicy:
inbound:
  - application: web

prometheus:
enabled: true

gcp:
sqlInstances:
  - name: db
→

Kubernetes + Cloud

  • Deployment
  • Service
  • Ingress + TLS
  • HPA (autoscaling)
  • NetworkPolicy
  • ServiceAccount + IAM
  • Secrets / Vault
  • Prometheus config
  • Logging pipeline
  • Cloud SQL + users + certs
  • Cloud SQL proxy
  • Observability stack
  • Det vi ser til venstre er det som skal til for å kjøre ut en applikasjon i NAIS
  • Du sier "kjør denne appen i NAIS, med disse egenskapene" og NAIS oversetter det til en hel haug med kubernetes og gcp-ressurser
  • Tilsvarende i Kubernetes + Cloud vil måtte håndtere
  • Service: Loadbalancer, intern DNS

⚖️ Kort oppsummert

  • En plattform som optimaliserer for fart, sikkerhet og standardisering
